1. The Summary of Actions
    Initial Audit: we cloned the repository and found the README.md file, but the password field was redacted with xxxxxxxxxx.
    Deep Inspection: we used git cat-file commit HEAD to look at the latest commit's metadata. The commit message "fix info leak" served as a major red flag that sensitive data had recently been removed.
    Historical Analysis: we ran git log -p to view the full history of changes (patches) for the repository.
    Extraction: By looking at the diff of the "fix info leak" commit, we identified the specific line that was removed (marked with -) which contained the original password, and the line that replaced it (marked with +).

2. Technical Understanding
In Linux, Git stores a snapshot of every single change ever made.
    Commits are Immutable: Once a commit is made, we cannot "edit" it to hide a mistake; we can only make a new commit that fixes it. The old version remains in the .git directory.
    The "Diff" Logic: The git log -p command compares two snapshots.
        Red (-): Content that existed in the previous version but was deleted.
        Green (+): Content that was added in the new version.
        By looking at what was "fixed," we found the "info leak."

3. Key Takeaway
The "Undo" history is a goldmine. In cybersecurity, simply overwriting a password and committing the change is one of the most common mistakes (often called "Secret Spillage"). To truly delete a secret from a Linux Git repo, we would have to "rewrite history" using complex commands like git filter-branch or bfg-repo-cleaner—simply changing the file isn't enough.
Linux Command	Purpose
git log -p	Shows the "Patch" (actual code changes) for every commit in history.
git cat-file -p [hash]	Pretty-prints the contents of any Git object (commit, tree, or blob).
git checkout [hash]	Rewinds our entire directory to a specific point in time.
